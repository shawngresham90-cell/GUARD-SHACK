# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repository is

A personal monorepo of small, independent web apps for Shawn Gresham's trucking businesses (Trucking Life With Shawn / Rosedale Transport / GoDataQ). Each top-level folder (or root-level `.html` file) is a separate app with its own audience and, in some cases, its own deployment. There is no shared framework, package manager, build step, linter, or test suite at the repo root — most apps are a single self-contained HTML file you open directly in a browser.

## Commands

There is nothing to build for the single-file HTML apps — edit the file and reload it in a browser.

**dashcam** (Node, ES modules):
```bash
cd dashcam
npm install
cp .env.example .env   # fill in XAI_API_KEY etc.
npm run dev            # node --watch server.js
npm start
```

**truckstops** — `truckstops/data.js` is generated; never edit it by hand. Edit the per-state files in `truckstops/data/*.json` (entries carry a `type`: `truck_stop`, `rest_area`, `service_plaza`, `weigh_station`, or `truck_wash`), then regenerate:
```bash
python3 truckstops/build-data.py
```

**Supabase edge functions** (Deno, in `shop/functions/`, `textblast/functions/`, and `truckstops/functions/`):
```bash
supabase functions deploy <function-name> --no-verify-jwt
supabase secrets set KEY=value ...
```
Each function's `index.ts` has a header comment listing exactly which secrets it needs. Functions are deployed with `--no-verify-jwt` and instead verify a shared secret themselves (e.g. `SHOP_NOTIFY_SECRET`, `BLAST_SECRET`).

**Supabase SQL** — the `.sql` files (`shop/*.sql`, `dataq/dataq-leads.sql`, `truckstops/tpc-locations.sql`) are meant to be pasted into the Supabase SQL editor. They are written to be additive and safe to re-run; keep that property when editing them.

## Architecture and conventions

**Single-file app pattern.** The default architecture here: one `.html` file containing all markup, CSS, and JS inline; state persisted in `localStorage`; no login; works offline. Some apps add a `.webmanifest` and a service worker (`shop/shop-sw.js`, `barbershop-sw.js`) to be installable PWAs. When changing an app, keep it self-contained — don't introduce build tooling or split files unless asked.

**Runtime config lives in a `CONFIG` block.** Apps that talk to Supabase (e.g. `dataq/index.html`, `shop/shop.html`) keep the Supabase URL, public anon key, and business settings (like `PAID_FILING_URL`/`PAID_FILING_PRICE`) in a clearly marked `CONFIG` object near the bottom of the HTML file. The anon key is public by design; RLS is the security boundary.

**Supabase security model.** Row-level security policies are written so the browser's anon key can only insert/update (e.g. leads), never read other users' data. Anything that reads sensitive data or holds credentials (Twilio, Gmail app passwords, partner API keys) lives in an edge function guarded by a shared secret. Don't move secrets into client-side HTML.

**Sync is best-effort.** Client apps treat Supabase pushes as optional: if offline or unreachable, data stays in `localStorage`, is flagged unsynced, and retries on next load. The app must never block on the network.

**Netlify hosting.** Individual apps are deployed as separate Netlify sites, configured with per-folder `_redirects` and `_headers` files (app shell served with `Cache-Control: no-store` so deploys take effect immediately). Only `dataq/` deploys automatically: `.github/workflows/deploy-godatq.yml` publishes `dataq/` to godatq.netlify.app on any push to `main` touching that folder (requires the `NETLIFY_AUTH_TOKEN` repo secret; the workflow fails loudly if it's missing). Other sites are deployed manually.

**Files to ignore, not imitate.** `*.zip` files at the root are legacy deploy artifacts (new ones are gitignored — don't commit zips). Files with `(1)` in the name (`shop/shop (1).html`, `shop/lineup (1).html`) are stale duplicates; the canonical files are the ones without the suffix. `shop/pm-forms/` holds fillable PDF/XLSX maintenance forms — binary assets, not code.

## Google Sheets (`spreadsheets/`)

`spreadsheets/registry.json` lists the Google Sheets Claude tracks — sheet ID, worksheet, columns, key column, and update rules. When asked to update a tracked sheet: resolve it in the registry first, read the target rows before writing (match by `key_column`, avoid duplicates), match existing column order and value style, report back with the sheet link, and update the registry if a sheet's columns have drifted. Full instructions are in `spreadsheets/README.md`.
