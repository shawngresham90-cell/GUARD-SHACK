# Story 1.2: App → cloud sync

## Status
InProgress — sync layer implemented in `../../index.html` (Cloud module +
Settings UI). Verifiable once a real Supabase project is connected.

## Story
**As a** shop owner,
**I want** the app to push my customers and appointments to the cloud and reflect
the send log back,
**so that** the scheduler sees current data and I can see what was sent.

## Acceptance Criteria
1. Creating/editing a customer or appointment in the app upserts it to Supabase.
2. The customer's `ok_to_text` consent flag syncs to the cloud.
3. The app pulls `message_log` entries and associates them with appointments.
4. Sync failures are non-fatal: the app keeps working locally and retries later.
5. No secrets are embedded in the client; only the public anon key + RLS.

## Tasks / Subtasks
- [ ] Add Supabase client to the app with the anon key (AC: 5)
- [ ] Upsert customers/appointments on save (AC: 1, 2)
- [ ] Pull and merge `message_log` for display (AC: 3)
- [ ] Queue+retry sync when offline/unreachable (AC: 4)

## Dev Notes
Reuses the existing "OK to text" toggle (already added to customer records) for
`ok_to_text`. Keep IDs stable so upserts don't duplicate. Depends on Story 1.1.

## Testing
- Edit a customer offline, reconnect, confirm it appears in Supabase.
- Confirm a `message_log` row shows on the right appointment in the app.
